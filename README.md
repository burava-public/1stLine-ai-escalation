
# 1stLine AI Escalation

This repository contains the example configuration and instructions for configuring client part of [1stLine by Burava](https://burava.com/1stline) AI Escalation.

**1stLine by Burava** is a Transparent alert and incident management proxy platform. It is designed to sit between your alert producer and destination, routing alerts as-is or with the modifications you choose, registering and parsing them, then processing escalation based on parsed data.

## Notice

Both 1stLine by Burava and 1stLine AI Escalation Controller are in **Public Preview**. All features are in active development and may change without prior notice. The same applies to the access to the feature itself, which may be revoked at any time. Please reach out us in `support@burava.com` for any questions or concerns regarding the feature. 

## What is 1stLine AI Escalation?

**1stLine by Burava** gives you the ability empower your escalation and automatic diagnosis with **AI assitants that you actually own**.

We do not sell you the inference, instead we give you the seamless integration where you can choose how deep you want to integrate 1stLine into your AI-assisted escalation workflow. 

Briefly, here is how it works:
1. You host a controller application in your private network that listens for escalation requests from 1stLine through the connector configuration.
2. You configure global, alert schema, or escalation chain scoped AI escalation rules triggered by conditions matched. Learn more about these terms in [Core Concepts](https://docs.burava.com/1stline/core-concepts).
3. For each rule, you define a set of enrichments of types: `Context`, `SKILL`, `MCP`, `Agent`, `CLI`, and `Note`.
4. When you execute `Escalate to AI` 1stLine uses the fingerprint and basic search to find similar alert instances and adds them to the escalation context.
5. All matched rules` enrichments + alert instance context are merged into a single escalation task and sent to your controller.
6. Controller dynamically scaffolds `opencode.json` with the generated based on the enrichments and pre-existing configuration.
7. MCP servers and CLI are added to allowed tools, but the **actual binaries must be pre-installed in the AI assistant server image**. 
8. SKILLS, Agents and Context are passed as links or file paths and a prompt with brief instructions on how to fetch their `.md` files and which authentication to apply. First two are also added to allowed tools in merged config.
9. Controller manages OpenCode server pool and keeps `active assignments + warm_opencode_servers` servers running. Then it watches for the response and suggested actions files that AI assitant should generate in the end of the diagnostic session.
10. Controller sends the response and suggested actions back to 1stLine, then 1stLine sends the response message to the same destination as the original alert and includes a one-click action to apply suggested actions by AI assistant.

AI assitant never receives original destination webhook URL and can't perform any alert action without you. It can only suggest action that you as a responder must execute.

**Note**: Giving too much access or tools to the AI assistant may allow it to overcome these guardrails. This can be desired or undesired behavior depending on your use case. In any case, most alert actions are not executable through direct API or MCP, so most likely AI assistant won't be able to resolve the firing alert without your involvement.

## Technical Notes:

- 1stLine AI Escalation Controller is a long pooling application that listens for incoming escalation requests from 1stLine and orchestrates AI assitant servers to handle those requests. This allows you to to host the controller in private network.
- Only [OpenCode](https://github.com/anomalyco/opencode) is currently supported as the AI assistant server. We do not own or claim ownership of the service. Any issues with the OpenCode server itself must be reported to their respective repository.
- The controller maintains an OpenCode server pool and reconciles it to `active assignments + warm_opencode_servers` (for example `warm_opencode_servers: 1` keeps one warm server while idle). For each assignment, pre-configured `opencode.json` file is merged with the configuration generated by your AI escalation rules.
- After assignment completion, controller runs `opencode export <sessionId>` and stores session export JSON at `.firstline/opencode-session-<sessionId>.json` in that assignment folder.
- Info logs (`LOG_LEVEL=INFO`, default) include lifecycle summary plus secret-redacted full task payload and merged OpenCode config. `LOG_LEVEL=DEBUG` additionally includes raw OpenCode API request/response and full diagnostic traces.
- You own your AI assistant server, its environment and inference. Controller image itself is built without OpenCode server included and you must build your own image with OpenCode server, required tools and MCP servers installed.

## Requirements:

- Active 1stLine by Burava subscription with AI Escalation feature enabled.
- Containerized environment to host the controller.
- Environment supporting long pooling applications.
- AI models provider supported by OpenCode. You can view the list of supported providers and models in the [OpenCode documentation](https://models.dev/?sort=provider-id&order=asc)

## What is included in this repository?

1. [Dockerfile](Dockerfile) - example Dockerfile to build the environment image for the controller. You must build your own image based on this example and include OpenCode server, required tools and MCP servers in it.
2. [docker-compose.yaml](example/docker/docker-compose.yml) - example docker compose configuration to run the controller locally. Not recommended for production use.
3. [k8s manifests](example/k8s) - example Kubernetes manifests to deploy the controller in a Kubernetes cluster.
4. [SKILL.md](example/opencode/skills/SKILL.md) - example SKILLs that we suggest to include into initial configuration of AI assistant server. If included, it will be always present in the merged configuration. It contains pre-flight instructions how to fetch context from enrichments, query 1stLine data and use it in the investigation, as well as how to perform diagnostics using generic tools like MCP and CLI.
5. [opencode.json](example/opencode/opencode.example.json) - example OpenCode configuration. 
6. [example-escalation-rules.json](example/1stLine/escalation-rules.json) - example escalation rules configuration that you can use as a reference when configuring your own rules in 1stLine. You can configure these rules in 1stLine UI or through API.
7. [controller.config.json](example/config/controller.config.example.json) - example controller configuration file.
